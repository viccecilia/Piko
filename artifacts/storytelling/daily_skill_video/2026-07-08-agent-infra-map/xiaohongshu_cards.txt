P3 Graphify
主标题：先把项目读成图
项目名：Graphify
一句话定位：把代码、文档、schema 映射成可查询知识图谱。
解决什么：大型项目不是一堆文件，而是依赖、调用、文档、schema 组成的关系网。
普通开发者能学什么：做 RAG 不一定只靠向量库，代码理解可以引入 evidence graph。
适合 / 不适合：适合代码库问答和架构理解；不适合简单全文搜索或不想维护索引的场景。
画面建议：GitHub / README 截图位 + 关系图解释。
读者价值：理解 Agent 为什么先要有“项目地图”。
状态：REAL_SCREENSHOT_REQUIRED

P4 Headroom
主标题：上下文不是垃圾桶
项目名：Headroom
一句话定位：在内容进入 LLM 前，压缩工具输出、日志、文件和 RAG chunks。
解决什么：Agent 工具输出越来越长，模型上下文被日志和碎片材料塞满。
普通开发者能学什么：上下文管理不是塞更多 token，而是筛选、压缩和保真边界。
适合 / 不适合：适合日志多、RAG chunk 多、工具输出长；不适合高精度原文引用场景。
画面建议：GitHub / README 截图位 + 日志漏斗压缩图。
读者价值：理解“压缩但不失真”会成为 Agent 基础设施问题。
状态：REAL_SCREENSHOT_REQUIRED

P5 OpenRath
主标题：Agent 需要运行时
项目名：OpenRath
一句话定位：面向 multi-agent / multi-session workflow 的 PyTorch-like runtime。
解决什么：多 Agent 不是多开几个聊天窗口，而是要管理 session、memory、tool 和 workflow。
普通开发者能学什么：把 Agent 过程看成可组织、可复盘、可替换的运行对象。
适合 / 不适合：适合多角色协作和实验型 runtime；不适合单轮聊天或成熟生产系统。
画面建议：GitHub / README 截图位 + Session/Memory/Tool/Workflow 结构图。
读者价值：理解 runtime 不是炫技，而是让复杂任务有形状。
状态：REAL_SCREENSHOT_REQUIRED

P6 Trigger.dev
主标题：长任务必须能恢复
项目名：Trigger.dev
一句话定位：用 TypeScript 构建 durable AI workflow 和 background tasks。
解决什么：AI workflow 会超时、失败、等待审批，不能只靠脚本从头跑到尾。
普通开发者能学什么：长任务要有队列、重试、日志、状态、人工介入和可观察性。
适合 / 不适合：适合后台任务和 AI 自动化流程；不适合只调用一次模型的小脚本。
画面建议：GitHub / README 截图位 + retry/queue/trace/approval 时间线。
读者价值：理解 Agent 工程会和任务队列、监控、审批系统汇合。
状态：REAL_SCREENSHOT_REQUIRED
