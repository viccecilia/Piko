# 别再乱学 Agent 了：GitHub 高星项目里，真正值得研究的只有这 5 类

如果你最近开始研究 AI Agent，大概率会遇到一个问题：

项目太多了。

LangGraph、CrewAI、AutoGen、LlamaIndex、Dify、Semantic Kernel、Pydantic AI、OpenAI Agents SDK……每一个看起来都很重要，每一个都有人说是未来。

但我看完一圈高星项目和主流 Agent 框架后，反而得出一个结论：

**普通人没必要什么都学。真正值得先搞懂的，是 5 类 Agent 能力。**

不是因为其他项目没用，而是因为这 5 类刚好覆盖了一个 Agent 系统从想法到落地的主干。

## 先说结论

如果你只想把 Agent 用起来，不是研究论文，也不是造轮子，可以先看这 5 类：

1. **LangGraph：负责流程和状态**
2. **LlamaIndex：负责资料和证据**
3. **CrewAI：负责多角色协作**
4. **AutoGen / AG2：负责多 Agent 对话和实验**
5. **Dify / OpenAI Agents SDK：负责应用化和工具调用**

它们不是互相替代的关系。

更像是一条生产线上的不同位置。

一个负责把流程跑稳，一个负责把资料找准，一个负责让不同角色分工，一个负责做多 Agent 实验，一个负责把能力做成真实可用的应用。

## 为什么不能只学一个？

因为 Agent 不是一个按钮。

真正能工作的 Agent 系统，至少要回答 6 个问题：

- 任务怎么拆？
- 状态怎么保存？
- 工具怎么调用？
- 证据从哪里来？
- 多个 Agent 怎么协作？
- 最后结果怎么验证？

很多人卡住，不是因为模型不够强，而是因为只学了其中一块。

比如只学工具调用，很快就会发现流程乱。

只学 RAG，很快就会发现不会安排任务。

只学多 Agent，很快就会发现每个 Agent 都在聊天，但没人负责验收。

所以我们不是要收藏 100 个项目，而是要看懂 5 个位置。

## 01 LangGraph：Agent 系统的流程骨架

LangGraph 最值得学的点，不是它能不能写一个聊天机器人。

而是它把 Agent 变成了一个可控制的工作流。

你可以理解成：

**LangGraph 解决的是“下一步该谁做、状态放在哪里、失败怎么重试”的问题。**

如果一个 Agent 系统开始变复杂，有搜索、有写作、有审核、有返工，它就不能只靠一段 prompt 硬撑。

它需要流程。

它需要状态。

它需要 Gate。

这就是 LangGraph 这一类框架的价值。

## 02 LlamaIndex：让 Agent 不再凭空回答

Agent 最大的问题之一，是看起来很聪明，但不知道它根据什么说话。

LlamaIndex 这类工具解决的是资料层。

它更像是 Agent 的证据仓库：

- 文档怎么切分
- 来源怎么索引
- 检索怎么做
- 证据怎么引用
- 回答怎么追溯

对 Piko 这种系统来说，这一层非常关键。

因为我们不是让 Agent 随便写文章，而是要知道：

它引用了什么来源？

这个结论有没有证据？

这句话能不能回到 source record？

没有证据的内容，不能直接发布。

## 03 CrewAI：把一个人脑袋里的分工拆出来

CrewAI 的核心价值，是让多 Agent 分工更容易理解。

比如一个内容系统里，可以有：

- Source Agent：找来源
- Evidence Agent：抽证据
- Ranking Agent：排优先级
- Writer Agent：写草稿
- Editor Agent：改结构
- Factcheck Agent：验事实

这不是为了“看起来像团队”。

而是因为真实工作本来就不是一个角色能做完。

CrewAI 这类框架的意义，是让角色、任务、协作关系更清楚。

但它也有风险：

如果没有验证和 Gate，多 Agent 很容易变成“多个 Agent 一起胡说”。

所以 CrewAI 适合用来表达分工，但不能替代证据链和验收系统。

## 04 AutoGen / AG2：多 Agent 实验场

AutoGen 这一类框架适合做什么？

适合做多 Agent 对话、协商、实验。

它的价值在于你可以观察：

- 不同 Agent 如何交换信息
- 一个 Agent 如何调用另一个 Agent
- 多轮讨论如何收敛
- 哪些协作模式有效，哪些只是热闹

如果你在探索“多个 Agent 怎么一起解决问题”，AutoGen/AG2 很值得研究。

但如果你的目标是稳定生产内容，它不应该单独成为终点。

你仍然需要工作流、证据、评分、验证和人工确认。

## 05 Dify / OpenAI Agents SDK：把 Agent 做成能用的应用

前面几类更偏“系统内部怎么跑”。

Dify 和 OpenAI Agents SDK 这一类更接近应用层。

它们解决的问题是：

- 怎么接工具
- 怎么接模型
- 怎么让用户使用
- 怎么把 Agent 能力封装成一个产品入口
- 怎么让开发者少写胶水代码

如果你只是想快速做一个可用的 Agent 应用，这一类非常有价值。

但如果你要做的是长期可升级系统，就不能只停在应用层。

你还要知道背后的流程、证据、记忆、验证和替换机制。

## 这 5 类组合起来，才像一个真正的 Agent 系统

可以把它们放成一张图：

```text
LangGraph：流程和状态
LlamaIndex：资料和证据
CrewAI：角色和协作
AutoGen/AG2：多 Agent 实验
Dify/OpenAI Agents SDK：应用和工具入口
```

这也是我现在做 Piko 的判断：

**不要训练模型。**

**不要完全自研 Agent runtime。**

**用成熟框架。**

**把业务规则、证据链、验证体系喂进去。**

**最后由人类确认。**

这条路线比“我自己从零写一个超级 Agent”更现实。

## 真正该学的不是项目名，而是位置

很多人学 Agent，会陷入项目收藏焦虑。

今天看 LangGraph，明天看 CrewAI，后天又看到一个新项目，感觉每个都不能错过。

但真正应该问的是：

这个项目在我的系统里负责哪个位置？

它替代了什么旧流程？

它能不能被测试？

它失败了能不能回滚？

它有没有证据链？

它会不会绕过人工确认？

如果回答不了这些问题，再高星也只是收藏夹里的一个链接。

## 对 Piko 来说，下一步是什么？

Piko 的目标不是只做游戏攻略。

游戏只是第一个领域。

真正的目标，是做成一个可插拔的多 Agent 协作模板：

- 今天可以分析游戏玩家痛点
- 明天可以分析 AI 工具趋势
- 后天可以分析本地生活问题
- 再后面可以分析任何结构化的信息需求

所以我们现在做的事情，是让 Piko 每天看开源世界里出现了什么新能力。

然后判断：

- 要不要进入能力图？
- 要不要替换旧 skill？
- 要不要做成一篇内容？
- 要不要生成一段视频？
- 要不要进入下一轮验证？

这才是“自动升级”的真正含义。

不是系统自己乱改代码。

而是系统每天发现新可能，提出候选，人类最后确认。

## 一句话总结

如果你想搞懂 AI Agent，不要一上来收藏 100 个项目。

先搞懂这 5 类能力：

1. 流程怎么跑
2. 证据怎么来
3. 角色怎么分
4. 协作怎么实验
5. 应用怎么落地

这 5 个位置看懂了，你再看任何 Agent 项目，都不会乱。
