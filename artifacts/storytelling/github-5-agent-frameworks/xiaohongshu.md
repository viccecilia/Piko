# 小红书图文卡片：别再乱学 Agent 了

## P1 封面

别再乱学 Agent 了  
GitHub 高星项目里  
真正该懂的是这 5 类

## P2 痛点

现在 Agent 项目太多了。  
今天 LangGraph，明天 CrewAI，后天 AutoGen。  
越收藏越焦虑。

问题不是项目少。  
问题是你不知道它们各自负责什么。

## P3 结论

普通人不用掌握 100 个 Agent 项目。  
先懂 5 类能力就够了：

流程  
证据  
分工  
实验  
应用

## P4 第一类：流程

代表：LangGraph  

它解决的是：  
下一步谁做？  
状态放哪里？  
失败怎么重试？  
什么时候进入 Gate？

一句话：  
它是 Agent 系统的流程骨架。

## P5 第二类：证据

代表：LlamaIndex  

它解决的是：  
资料怎么索引？  
证据怎么引用？  
回答怎么追溯？  
哪些话不能乱说？

一句话：  
它让 Agent 不再凭空回答。

## P6 第三类：分工

代表：CrewAI  

它解决的是：  
谁找资料？  
谁写草稿？  
谁做审核？  
谁负责返工？

一句话：  
它把一个人的复杂工作拆成一组角色。

## P7 第四类：实验

代表：AutoGen / AG2  

它适合观察：  
多个 Agent 怎么对话  
怎么协商  
怎么互相调用  
怎么收敛到结果

一句话：  
它是多 Agent 协作的实验场。

## P8 第五类：应用

代表：Dify / OpenAI Agents SDK  

它解决的是：  
怎么接工具  
怎么接模型  
怎么做成用户能用的入口  
怎么少写胶水代码

一句话：  
它让 Agent 从 demo 变成应用。

## P9 最关键的一张图

不要问：哪个 Agent 最强？

要问：  
它在系统里负责哪个位置？

流程：LangGraph  
证据：LlamaIndex  
分工：CrewAI  
实验：AutoGen/AG2  
应用：Dify/OpenAI Agents SDK

## P10 对 Piko 的启发

我的判断是：

不训练模型。  
不完全自研 Agent runtime。  
用成熟框架。  
把业务规则、证据链、验证体系喂进去。  
最后由人类确认。

## P11 总结

真正值得学的不是项目名。  
而是 Agent 系统的 5 个位置。

看懂这 5 个位置，  
你再看任何新 Agent 项目，  
都不会乱。
