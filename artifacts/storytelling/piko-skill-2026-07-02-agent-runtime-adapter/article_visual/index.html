别急着接入 Agent 框架，先给 Piko 加一道适配器边界

Piko Daily Skill · 2026-07-02 · AgentRuntimeAdapter boundary

今天这个选题不是介绍某一个新 AI 框架。它要解决的是：Piko 怎么学习 LangGraph、OpenAI Agents SDK、LlamaIndex 这类能力，但不把系统控制权交出去。

01 开场判断：先定边界，再谈接入

以前研究 agent 框架，容易走向两个极端：一是只做调研，能力进不了 pipeline；二是直接接入运行时，把来源策略、审核闸门、发布限制都暴露给新框架。

所以这次的结论很直接：Piko 不应该先安装框架，而应该先定义外部框架进入 Piko 的门禁。

02 新能力：AgentRuntimeAdapter boundary

AgentRuntimeAdapter 不是新的系统主控，而是一个可测、可审计、可替换的接口层。外部 agent 框架只能通过它执行边界清楚的任务，并返回结构化结果、证据卡引用和验证输入。

run_task一次只执行一个有输入和输出的任务。
tool_policy声明允许哪些工具和外部副作用。
state_contract只能读写 Piko pipeline 允许的状态字段。
evidence_contract返回证据卡引用，不搬运长原文。
verification_contract必须输出验证材料，不能绕过 Piko verify。
图 2：五个 contract 是这个适配器的核心。

03 快速判断：值得做，但不直接替换 pipeline

今天的判断是：这个能力值得进入 CAP，但只应该做增强层。它不应该立刻替换 article_pipeline。

更合理的顺序是：先做 offline fixture，再跑 adapter contract tests，确认新 runtime 不能越权，最后再考虑真实框架接入。

05 实操步骤

  1. 列出候选框架：LangGraph、OpenAI Agents SDK、LlamaIndex。
  2. 抽出共同能力：任务执行、工具策略、状态读写、证据返回、验证材料。
  3. 先写最小 adapter interface，不默认接 LLM，不默认发 API。
  4. 用 offline fixture 跑 contract tests，测试外部 runtime 不能越权读写状态。
  5. 把结果交给 CAP：augment first，replace later。
adapter.run_task(input)

  - returns evidence card refs
  - emits verification inputs

06 真实限制

这个方案目前还是 proposal，不是已上线功能。风险中等：adapter 写得太宽会变成隐藏运行时；写得太窄又看不到真实框架价值。

07 总结和行动

Piko 真正要学的不是某个框架 API，而是让所有外部 agent 框架都必须遵守同一套边界。

  • 先做 AgentRuntimeAdapter 的离线 fixture。
  • 再补 adapter contract tests。
  • 确认 verify、source policy、publishing disabled 三个闸门都不可绕过。
