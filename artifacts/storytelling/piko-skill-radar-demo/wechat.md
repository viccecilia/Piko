# 我给 Piko 加了一个“每日 Skill 雷达”：让系统每天自己寻找更好的 Agent 能力

封面短句：

AI 系统真正的升级，不是多接几个工具。  
而是知道什么时候该替换工具。

## 开头

以前我们做一个 AI 项目，经常会遇到一个问题：

今天用了一个 Agent 框架，明天 GitHub 上又冒出一个更强的。  
今天接了一个 Skill，过几天发现另一个 Skill 更稳定、更便宜、更适合当前任务。

如果每次都靠人手动搜索、判断、替换，这个系统永远长不大。

所以我给 Piko 设计了一套新的能力：

**每日 Skill 雷达。**

它不是让系统自动乱升级。  
它要做的是：每天扫描高价值开源项目和可用 Skill，把值得借鉴的能力放进能力地图，再交给人类做最后确认。

## 01 它解决的不是“找工具”，而是“判断工具值不值得进入系统”

搜索 GitHub 并不难。

难的是这几个问题：

- 这个项目真的适合 Piko 吗？
- 它解决的是我们已经有的能力，还是新的缺口？
- 它能替换现有 Agent，还是只适合做参考？
- 它的 license、安全边界、维护状态可靠吗？
- 引入它以后，会不会破坏已经验证通过的工作流？

这才是 Skill 雷达真正要解决的问题。

## 02 它每天会看什么

Piko 会重点关注这些方向：

- Agent framework
- Workflow orchestration
- RAG / evidence indexing
- Plugin architecture
- Connector framework
- Evaluation / guardrail
- Automation
- Operator dashboard
- Self-improvement system

简单说，就是所有能让 Piko 更自动、更稳、更可插拔的能力。

## 03 它不会自动改代码

这是最重要的一点。

Skill 雷达不会看到一个高星项目，就立刻把代码塞进 Piko。

它只会输出：

- 项目是什么
- 星标数和活跃度
- 它的核心设计模式
- Piko 能借鉴什么
- 风险在哪里
- 是否值得进入 worker/verify 队列

真正升级仍然必须经过：

```text
扫描日报
-> 升级提案
-> Worker 实现
-> Verify 验证
-> 人类最终确认
```

## 04 为什么这对 Piko 很关键

Piko 的目标不是做一个固定的游戏攻略工具。

它要变成一个可插拔系统：

- 今天做游戏玩家痛点
- 明天做电商差评分析
- 后天做 SaaS 用户反馈
- 再以后做旅游、本地生活、知识库

如果底层能力不能持续进化，Piko 很快就会变成一个过时的工具箱。

所以能力地图必须是活的。

有更好的 Skill，就进入候选。  
现有 Skill 不稳定，就降级。  
某个能力风险太大，就拒绝。  
真正要替换时，再走验证流程。

## 05 它最终想做到什么

理想状态下，Piko 会自己完成这些事：

1. 发现当前任务。
2. 选择合适 Agent 和 Skill。
3. 调用工具完成工作。
4. 生成结果和证据。
5. 自己跑测试和验证。
6. 把需要人类确认的部分交出来。

人类不再盯着每个步骤。  
人类只做最后的判断。

## 06 现在还差什么

现在这套能力还不能直接全自动替换系统组件。

原因很简单：

- 外部项目可能有 license 风险
- 不同 Agent 框架的抽象不一样
- 有些 Skill 需要 credentials
- 有些工具会产生费用
- 自动替换可能破坏已验证流程

所以 Piko 目前只允许它做“学习和提案”，不允许它偷偷升级。

## 总结

每日 Skill 雷达的价值，不是多找几个工具。

而是让 Piko 拥有一种能力：

**持续发现更好的能力，但保持升级可控。**

这也是我认为一个长期 AI 系统必须具备的东西。

不是一次性搭完。  
而是每天变好一点。
